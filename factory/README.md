# 🏭 問題工場（Mock Factory）— ハブ

このリポジトリ（`master`）を **正本（メイン）** とし、作問・品質管理を行う複数の
「スレッド（作業の流れ）」を**ブランチ＋定期セッション(cron)**で回し、**PR で `master` に合流**させる。
PR のマージ＝**人（先生）のレビュー関門**。自動化と人の確認を両立させるのが狙い。

```
master（正本：模試/単語/英検 + 採点エンジン + 検証ハーネス + GAS + 管理ページ）
├─ qa/loop       ① 品質チェックを定期巡回 → 不具合を自動修正/レポート → PR
├─ gen/predict   ② 年間予定から第3回を予測 → 模試を先行生成 → PR
└─ gen/okayama   ③ 岡山入試を収集・分析（reference-only）→ 傾向準拠の新作模試 → PR
   各ブランチは cron で自走 →（CIの品質ゲート通過）→（人がレビュー）→ master へマージ
```

---

## 品質ゲート（全スレ共通の土台）

すべての模試データは **`npm run check`** を通る必要がある。中身:
- 全模試を**実エンジンで採点 → 全問正解で 100/100** になるか（出題と採点の整合）
- 設問の構造チェック（`mcq`/`bankpick` の answer 索引、`wordorder` の語が answer 内にあるか 等）
- 単語データのサニティ（「↑の複数」等の**相対参照の訳**・空訳・件数）

- ローカル: `npm install && npm run check`
- CI: `.github/workflows/exam-check.yml` が **push / PR ごとに自動実行**（ブランチの PR はここが緑でないとマージしない）
- 実装: `tools/check_exams.mjs`

> この品質ゲートは実際にバグを検出してきた（`feet=↑の複数` / `English=PE番目` / chu3_2 大問4 の Saki 連続セリフ 等）。①はこれを **AIレビュー**で拡張する。

## ブランチ規約

| ブランチ | スレッド | 役割 |
|---|---|---|
| `master` | — | 正本。公開される模試・単語・英検・GAS。 |
| `qa/loop` | ① | 品質チェック巡回。修正は小さくPR。 |
| `gen/predict` | ② | 年間予定ベースの予測模試生成。 |
| `gen/okayama` | ③ | 岡山入試 reference-only の新作模試生成。 |

- 各スレは**自分のブランチ**で作業し、`master` への **PR でリンク**する。
- PR タイトルに `[qa]` `[gen:predict]` `[gen:okayama]` を付ける。
- 生成・修正は**小さく刻んで** PR（レビューしやすく、巻き戻しやすい）。

---

## スレッド仕様

### ① qa/loop — 品質チェックをぐるぐる回す（実現度◎）
1. `npm run check`（採点・構造・単語）
2. AIレビュー（大問ごとに並行）：論理・話者の連続・訳の文脈依存・難易度・誤答選択肢の妥当性
3. **反証検証**（findings を別エージェントで偽検出つぶし）
4. 安全な不具合は自動修正、判断が要るものはレポート → PR
- 状態：**骨組み完了**（`npm run check`＋CI）。AIレビュー段は次フェーズで追加。

### ② gen/predict — 年間予定から予測モシ（実現度○・要入力）
- 入力：**年間指導計画**（`factory/inputs/annual_plan.*` に置く想定）＋過去回の傾向
- 出力：既存フォーマットの新作模試（`mogi/data/`）→ `npm run check` 100点 → PR
- 状態：**待ち**（年間予定の提供が必要）。

### ③ gen/okayama — 岡山入試 過去問模試（実現度△・reference-only）
- 方針：**reference-only**。傾向・難易度・配点だけを参照し、**オリジナル新作**を量産（著作権安全・「内容は新規」方針と整合）。
- ⚠️ 過去問の**verbatim 収録は権利確認が前提**。本工場の既定では行わない。
- 収集：Web 取得は出典の公開状況・接続可否の検証が必要。
- 状態：**計画**（reference-only で着手予定）。

## 自走（cron）の仕組み

- **無料の自走**：CI（GitHub Actions）が push/PR ごとに品質ゲートを回す＝常時稼働の検査。
- **AIの自走**：定期セッション（cron）が各スレの作業（AIレビュー/生成）を実行 → PR。
  - 周期は「毎晩／毎週」などのカデンス（24時間ベタ稼働ではない）。
  - ディスパッチ手順は `factory/dispatch.md` を参照（このハブが指示書）。
- 生成物は必ず PR を経由し、**人がレビューしてからマージ**。
