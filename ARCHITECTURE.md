## 1. これは何か

中学生向けの英語学習アプリ集（山田先生）。**3つの学習アプリ＋マイページ＋先生用管理ページ**で構成。

| 観点 | 設計判断 |
|---|---|
| 動作環境 | ブラウザだけで完結。インストール不要、スマホ可。 |
| 保存 | 生徒の記録は **localStorage**（端末ローカル）。サーバー必須にしない。 |
| 集約 | 先生に届ける時だけ **GAS 経由でスプレッドシート**に集約（1人1行 upsert）。 |
| バックエンド | **GAS ウェブアプリ1本**（`tools/score_gas.gs`）。CORSプリフライト回避のため `text/plain` POST。 |
| 人の関与 | 単元テストの開始/終了・学習方針は**先生がゲート制御**（teacher-in-the-loop）。 |
| 品質保証 | 全コンテンツが `npm run check`（実エンジン採点＋構造検査）を通る。CI＋自走スレ（問題工場）。 |

**設計の背骨**：「生徒＝ローカルで練習・記録」「先生＝集約された成績を見る・方針を出す」「コンテンツ＝機械検証で品質担保」。この3者を疎結合につなぐのが GAS と localStorage。

## 4. データモデル（最重要）

新アルゴリズムが読む「センサー入力」と書く「介入先」は、ほぼすべてここにある。

### 4a. localStorage（生徒のローカル記録＝同一オリジンで全アプリ共有）

| キー | 形 | 意味 | 書く人 |
|---|---|---|---|
| `tango_v2` | `{cleared:{id:ts}, settings:{retry,sound,order}}` | 単語アプリ：打ち込めた語ID集合＋設定 | words |
| `mogi_best_v1` | `{examId:{title,best,last,total,ts}}` | 模試：各回の最高/直近点（100点満点） | engine |
| `shuku_v1` | `{c2:{best,max,count,last}, c3s2:{...}}` | 習熟度単語テスト：最高点・受験回数 | vocab_*.html |
| `mado_year` / `mado_num` / `mado_name` | string | 本人情報（学年1〜3／番号＝半角／名前） | me, exam, eiken |
| `mado_policy` | string | 先生の学習方針コード（送信時にサーバーから受信しキャッシュ） | me |
| `mado_snap_v1` | `{mBest,vBest,act,day}` | 前回スナップショット（ほめ言葉の差分計算用） | me |
| `mado_bestfull_v1` / `mado_bestquick_v1` | number | 英検：本番/クイックの最高レベル座標(0〜8) | eiken |
| `mado_levelavg_v1` | `{sum,count}` | 英検：平均レベルの目安 | eiken |
| `mado_loop_lasttest_v1` | `{grade,pct,pass}` | 英検：直近の判定テスト結果 | eiken |

> **重要**：localStorage はオリジン単位。3アプリ＋マイページは**同じ記録を共有**する。
> 「打ち込めた語数」「模試ベスト」「習熟度%」「英検レベル」が、生徒1人の状態ベクトル。

### 4b. GAS スプレッドシート（先生の集約ビュー）

| シート | 役割 | 主キー | 更新方式 |
|---|---|---|---|
| `成績まとめ` | 3アプリ最新/最高を**1人1行** | 学年/番号 | upsert（`max`列は下げない・他は最新上書き） |
| `英検テスト履歴` | 英検の**1回ごと**追記 | — | append |
| `単元管理` | 各単元テストのゲート状態 | 試験ID | 状態(開/閉)・セッション・提出数 |
| `単元テスト記録` | 単元テスト提出を1回ずつ | セッション+試験+学年+番号 | append（同一で重複拒否） |
| `設定` | 学習方針コード（B2） | — | 先生がadminから設定 |

`成績まとめ` の列定義は `score_gas.gs` の `SUMMARY_COLS`（`{key, head, max}`）が単一の真実。
マイページの `buildPayload()` のキーと1対1で対応する（＝**ここに列を足せば集約に乗る**）。

### 4c. コンテンツのスキーマ

**単語**（`words/data/words.js`、`build_words.py` が自動生成）:
```js
window.WORD_META = {total, target, posOrder, basicCount, extendedCount,
                    basicByPos, extendedByPos, droppedCount};
window.WORDS = [{ id, w /*英語*/, j /*訳*/, p /*品詞*/, m:"b"|"e" /*基本/拡張*/,
                  g:1|2|3 /*学年*/, alt?:[/*別表記も正解*/] }, ...];
```

**模試/単元テスト**（`mogi/data/*.js`、`engine.js` が描画＆採点）:
```js
window.EXAM = {
  id, title, unit?:true /*単元テスト＝mogi_best保存せずGAS提出*/,
  sections:[{
    no, title, lead?,
    courses?:[{name,intro?,items}],          // 大問6 X/Yコース選択
    groups?:[{ intro?, script?/*放送文*/, sceneNote?, flyer?,
               passage?, passageEn?, note?, items:[ITEM] }]
  }]
};
// ITEM 型（自動採点）:
//  mcq      {stem,choices,answer:idx,pt,label}
//  mcqMulti {answer:[idx,...]}
//  fill     {answers:[...],hint}      ← 表記ゆれ正規化で照合
//  bankpick {bank,answer:idx}         ← 共通選択肢
//  wordorder{words:[...],answer:"完成文"}  ← タップ並べかえ
//  en / jp  {model,minWords?}         ← 模範解答提示・自己採点
```

## 5. 主要フロー

**(1) 採点（模試/単元）** — `engine.js`
```
MockExam.render(EXAM,#quiz) → 生徒が解答 → MockExam.gradeAll()
  → 各ITEMを採点・○✕表示 → 合計/満点を .scorebar に表示
  → window.MockExam.last={score,total,pct,id}
  → EXAM.id かつ !EXAM.unit のとき mogi_best_v1 に最高点保存
  → 一番上へスクロール
```

**(2) 単元テストのゲート** — `admin` ⇄ `exam.html` ⇄ GAS
```
先生: admin で PIN＋[スタート] → action:"gate"{open:true}
   → GAS が 単元管理 にセッション "S+日時" を発行（LockService）
生徒: exam.html?id=c2u1 が 5秒ごとに action:"status" でポーリング
   → 受付中になったら採点UI解放 → 提出 kind:"unittest"
   → GAS: ゲート開＋同一(セッション+試験+学年+番号)未提出のときだけ記録（1人1回）
```

**(3) 成績集約・送信** — `me` → GAS
```
マイページを開く → localStorage 全読み取り → buildPayload()
  → kind:"summary" を POST → GAS が 成績まとめ に upsert
  → 戻り値で現在の学習方針 policy を受信 → コメント更新
  ※ 習熟度単語は 80%以上のときだけ正式点を送る（回数は常に送る）＝水増し防止
```

**(4) 学習方針 → コメント** — `admin` → GAS → `me`
```
先生: admin で方針を選び保存（action:"setpolicy"｜PIN）→ 設定シート B2
生徒: マイページ送信の戻り or action:"policy" で方針を取得
   → guidance(policy, records) が「次の一手」を1行表示
```

## 6. 育成エンジン

> **ここが既存の「学習者育成ロジック」。新アルゴリズムはこの考え方を継承・自動化する。**

マイページ（`me/index.html`）の `📣 きょうのミッション` カードは、2行で生徒を動かす：

- **ほめ1行** `praise(R, snap)`：前回スナップショット `mado_snap_v1` との**差分**で文面を選ぶ
  （模試が上がった→「○点アップ！」、活動が増えた→「挑戦してますね！」、下がっても→「続けててえらい」…）。
- **指針1行** `guidance(policy, R)`：先生の**方針コード**に応じて「次にやること」を出す。

方針コード（`POLICIES` / admin の選択肢）:
```
kihon（基本単語） / shuku（習熟度単語） / kihon_shuku /
moshi（模試） / eiken（英検） / balance（バランス）
```
`guidance()` は方針ごとに、生徒の状態 `R`（模試ベスト・習熟度%・基本語進捗・活動量…）を見て
「未達なら優先タスク／達成なら次ステージ」を分岐する。

**＝これは小さな制御ループ**：
```
状態を測る(records) → 方針(teacher) → 介入(praise+guidance) → 次回また測る
```
現状は方針が**手動**・分岐が**ルールベース**。新アルゴリズムはこのループを
**データ駆動・適応的**にする余地が大きい（→ §8）。

`records()` が返す状態ベクトル（育成アルゴリズムの入力に直結）:
```
mBest, mDone,                  // 模試 最高点・挑戦回数
vC2, vC3, vBest, vC2n, vC3n,   // 習熟度単語 %・回数
wBasic, wBasicT, wAll,         // 基本語 進捗・総語数
act,                           // 活動量（模試回数+単語テスト回数+単語+英検）
day                            // 今日の日付（連続性判定）
```

## 7. 品質ゲート・問題工場

**品質ゲート** `tools/check_exams.mjs`（`npm run check`）:
1. 全11模試を**実エンジンで採点 → 全問正解で 100/100** になるか（出題と採点の整合）
2. 構造検査（mcq/bankpick の answer 索引、wordorder の語が answer 内にあるか 等）
3. 単語サニティ（相対参照の訳・空訳・**答えバレ**＝英単語が訳に露出）
- CI（`exam-check.yml`）が push/PR/毎週/手動で実行。**PRはここが緑でないとマージしない**。

**問題工場（ツリー）** `factory/`:
```
master（正本）
├─ qa/loop      ① 品質チェックを巡回 → AIレビュー＋反証検証 → 修正/レポート → PR
├─ gen/predict  ② 年間予定から予測模試を先行生成 → PR（要：annual_plan 入力）
└─ gen/okayama  ③ 岡山入試 reference-only（傾向だけ参照しオリジナル新作）→ PR
```
自走は2層：**A=GitHub Actions（持続・無料の検査役）／B=Claude Code on web のスケジュール（作問・レビュー）**。
PRのマージ＝**人（先生）のレビュー関門**。

## 8. 拡張ポイント

> **「プロンプトを使いこなす学習者育成アルゴリズム」を別スレで作るときの接続マップ。**
> 既存の疎結合構造をそのまま使えば、最小の改変で「新しい学習軸＋その育成ループ」を足せる。

> ⚠️ 混同注意：基本アプリ側の **挑戦モード（`challenge/`）の「英検二次プロンプト生成」は別物**。
> あれは固定テンプレをコピペするだけの **“面接の練習用”** で、データ収集も評価もしない（基本レーン内）。
> §8 が扱うのは **プロンプト運用スキルそのものを育てる AIモード**（収集・分析・蒸留つき／§8.0 の2レーン分離）で、目的も構造も異なる。

このアプリは既に **「学習サーフェス → ローカル記録 → 集約 → 育成コメント」** の型を持っている。
新アルゴリズム＝**プロンプト運用スキルという第4の学習軸**を、同じ型に乗せるのが素直。

### 8.0 必須要件：いつでも切り離せる「2レーン分離」🔌

> プロンプトモードは基本アプリと**疎結合のまま並走**させ、**いつでも丸ごと切り離せる**こと。
> そのため次の2点を設計の前提とする（＝既存の成績システムには混ぜない）。**この章の他の提案より優先。**

**(1) 接続 ON/OFF（管理者ボタン）**
- 先生用 `admin/` に「**プロンプト連携 ON/OFF**」トグルを置く（PIN必須・`gate`/`policy` と同じ流儀）。
- 状態はフラグとして保存（例：プロンプト専用GASの `設定` シート、または `prompt_enabled`）。
- **ON のとき**：トップ `index.html` にプロンプトのカードを出し、マイページにプロンプト育成欄を表示。
- **OFF のとき**：基本アプリ（`words`/`mogi`/`eiken`/`me`）は**今日と完全に同じ挙動**。プロンプト関連UIは出さず・通信もしない。
- **フェイルセーフ**：フラグ取得に失敗したら **OFF 扱い**（プロンプト側の障害が基本アプリを止めない）。

**(2) プロンプト学習データは「別スプレッドシート」— 収集先はプロンプト設計スレッドが所有**
- 成績管理スプシ（`成績まとめ` 等）には**一切書かない**。`SUMMARY_COLS` は**不変**。
- **専用スプシ＋専用GASは、本リポジトリではなく「プロンプト設計スレッド」側で作成・所有する**
  （データの**収集・分析・蒸留**をあちらで行うため）。本リポジトリはスキーマも分析も持たない。
- **本リポジトリ側の責務は2つだけ**：
  ① 送信先URL定数 `PROMPT_GAS_URL`（プロンプトスレが用意したURLを**後で差すだけ**。未設定なら送信しない＝OFF相当）、
  ② admin の ON/OFF トグル。
- フロントは結果を `PROMPT_GAS_URL` に投げる**薄いエミッタ**に徹する。`score_gas.gs`／`SUMMARY_GAS_URL` は触らない。

> 「切り離す」操作は、**(a) admin で OFF にする（ソフト）** ／
> **(b) `prompt/` フォルダ＋`PROMPT_GAS_URL` 定数を消す（物理）** のどちらでも、
> 成績システムに痕跡を残さず完了できる。逆に言えば、**この2レーンが交わる実装をしないこと**が制約。

```
   基本レーン（今のまま・不変）            プロンプトレーン（新設・隔離）
   words/mogi/eiken/me ─▶ score_gas ─▶    prompt/ ─▶ PROMPT_GAS_URL ─▶
   成績まとめ等のスプシ                     プロンプト専用スプシ（別ID）
        │                                        │
        └──────── admin（PIN）───────────────────┘
                   ├ 既存：ゲート/方針
                   └ 追加：プロンプト連携 ON/OFF（このトグルが唯一の交差点）
```

### 8.1 接続の5点セット（§8.0 の分離を守ったうえで）

| # | 足す場所 | 既存のお手本 | 内容 |
|---|---|---|---|
| 1 | **学習サーフェス** | `words/`（独立アプリ） | プロンプト練習の出題・評価UI。**独立した `prompt/` フォルダ**にし、ON/OFFで丸ごと隠せる/消せる単位にする。 |
| 2 | **ローカル記録** | `tango_v2` / `shuku_v1` | 例：`prompt_v1 = {best, count, last, skills:{明確さ,文脈,制約,反復改善…}}`（既存キーとは別キー）。 |
| 3 | **集約（別スプシ）** | `handleSummary`／`SUMMARY_COLS` | **プロンプト設計スレが所有する専用GAS（別URL・別ID）**へ**送るだけ**。成績まとめは触らない。スキーマ・分析・蒸留はあちら。 |
| 4 | **育成コメント** | `POLICIES` ＋ `guidance()` | 方針 `prompt` を追加し分岐を1本。**ただしフラグ ON のときだけ**マイページに表示。 |
| 5 | **評価ロジック** | `engine.js` の自動採点 / `check_exams.mjs` | 固定正解で測りにくい → **LLM評価（Claude API）をプロンプト専用GAS経由**で（§8.3）。 |

### 8.2 「育成アルゴリズム」を載せる場所
§6 の制御ループ（測る→方針→介入→再測定）が**そのままアルゴリズムの宿主**になる。
- **入力（センサー）**：`records()` の状態ベクトル＋新キー `prompt_v1` のスキル内訳。
- **方針（policy）**：今は先生が手動選択。新アルゴリズムは
  状態から**次に伸ばすべきスキルを自動推定**して動的方針を出せる（適応的カリキュラム）。
- **介入（intervention）**：`praise()`/`guidance()` の1〜2行。
  プロンプト軸では「より良い言い換え例」「不足している制約の指摘」を返すなど、LLM生成の余地。
- **測定（measure）**：次回送信の差分（`mado_snap_v1` 同様のスナップショット）で効果を見る。

### 8.3 プロンプトスキルの自動評価（LLM）— 設計の選択肢
固定正解がない領域なので、評価には外部LLMが要る。CORS回避の既存流儀（`text/plain` POST）に合わせ、選択肢は：
- **(A) プロンプト専用GASを中継（推奨）**：**`score_gas.gs` ではなく**、プロンプト設計スレ側の `prompt_gas.gs`（別デプロイ・別ID）に `action:"prompt_eval"` を置き、そこから Claude API を叩いて採点JSONを返す（鍵はGAS側に隠す）。§8.0 の分離を保てる。
- **(B) 別サーバーレス**：評価専用の小さなエンドポイントを別途用意（GASのレート/制限を避けたい場合）。これも成績GASとは別系統。
- いずれも**ルーブリック固定**（明確さ/文脈/制約/出力形式/反復改善 などの軸）にして、
  返り値を `prompt_v1.skills` に保存 → **プロンプト専用スプシ**へ集約・マイページのコメントへ流す（フラグON時のみ）。

### 8.4 品質ゲートとの整合
新コンテンツ（プロンプト課題・ルーブリック）も `npm run check` に**検査を1本足す**のが工場流儀。
固定正解がないものは「ルーブリックJSONの構造検査」「サンプル解答がスコア境界を跨ぐか」等で代替検証する。

## 9. デプロイ・運用

| 対象 | 方法 | 注意点 |
|---|---|---|
| 静的アプリ | `master` への push で **GitHub Pages 自動反映**（`.nojekyll`） | ビルド不要。`*.xlsx`・ツール・依存は公開から除外。 |
| GAS | コードを貼替 → **[デプロイを管理]→✏️→「新バージョン」** | **保存だけでは反映されない**。URLは不変。`SPREADSHEET_ID` は貼替で消えるので再設定。 |
| 認証 | `TEACHER_PIN`（既定 `"PIN"`／要変更）＝ admin のゲート/方針 | adminは noindex。URLを生徒に教えない。 |
| 版確認 | `GAS_VERSION`（`"unit-gate-2"`）を admin が表示し stale 検出 | 旧版でもゲート対応なら警告しない（`gateOK()`判定）。 |
| 名簿 | `CLASS_SIZE = {"2":40,"3":40}`（未提出者算出の母数） | クラス人数に合わせて変更。 |
| 初期化 | `refreshData()` を手動実行で見出し統一＋古い記録消去 | 学年表記の移行などで使用。 |

GAS URL は2系統：**me/exam/admin = `AKfycbySJy2…`（score_gas）／eiken = 別URL**（同一スプレッドシート）。
今後追加する**プロンプトモードは第3の独立系統**。ただし**専用GAS・専用スプシ（別ID）はプロンプト設計スレッド側で構築・所有**し（収集/分析/蒸留はあちらで実施）、本リポジトリは送信先 `PROMPT_GAS_URL`（後差し・未設定なら送らない）と admin トグルだけを持つ（§8.0）。

## 10. 守るべき制約・原則

- **アプリにビルド工程を持ち込まない**（静的HTML/JSのまま。GitHub Pagesで即公開）。
- **CORSプリフライト回避**：GASへのPOSTは必ず `Content-Type: text/plain`、本文はJSON文字列。
- **localStorageはオリジン単位**：3アプリ＋マイページで記録を共有する前提を壊さない。
- **学年・番号は半角限定**（`han()`）：スプレッドシートのソート/突合のため。
- **習熟度は80%以上だけ正式送信**（水増し防止）。記録系は `max` で下げない。
- **teacher-in-the-loop**：単元テストの開閉・学習方針は先生が制御（自動で走らせない）。
- **プロンプトモードは2レーン分離（§8.0）**：基本アプリと疎結合のまま並走させ、
  ① admin の「接続 ON/OFF」でいつでも切り離せる（OFFなら基本アプリは今日と同一・フェイルセーフOFF）、
  ② プロンプト学習データは**専用GAS・専用スプシ**に隔離（`成績まとめ`/`SUMMARY_COLS` には混ぜない）。
- **コンテンツは必ず `npm run check` を通す**：壊れた出題はCIで止める。
- **生成・改変は小さくPR → 人がレビューしてマージ**（問題工場の鉄則。`master`直push しない）。
